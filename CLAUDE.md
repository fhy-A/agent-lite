# Code Claude Code 协作入口

开始任何开发、测试或评审任务前，先阅读：

1. `CHANGELOG.md`：已完成的重要改动、产品决策与验证结果。
2. `TODO.md`：未完成事项、优先级与后续计划。

任务结束前，将已完成的重要改动写入 `CHANGELOG.md`，将新发现或仍未完成的工作写入 `TODO.md`。新增日志必须写明本地具体时间戳，格式为 `YYYY-MM-DD HH:mm`；既有未带时间的历史记录无需追溯修改。不要在本文件复制项目状态，避免形成多个事实源。

## 分阶段协作与交付

1. **一次只推进一个阶段**：先从 `TODO.md` 和现有实现确认当前阶段的目标、边界与验收条件；阶段完成前不混入下一阶段功能。发现旁支问题时，只有会阻塞当前验收才在本阶段修复，否则记入 `TODO.md`。
2. **先验证，后提交**：每个阶段至少运行与改动直接相关的定向测试，并按风险决定是否运行完整回归、语法检查、构建或 `git diff --check`。测试失败时先修复并重新验证，不提交已知失败的阶段。
3. **区分自动与人工验收**：能够由自动化测试充分覆盖的阶段，由协作 Agent 自行完成验证、日志和提交；涉及视觉效果、流式交互、焦点、时序、浏览器差异或其他需要观察的行为时，向用户提供简短明确的测试步骤和通过标准，等待用户确认正常后再提交。
4. **人工测试发现问题就暂停收口**：用户反馈异常时，保持当前阶段未提交，先定位、修复并重新给出验证方式；只有用户确认或等价自动化证据通过后，才将阶段标记完成。
5. **每阶段独立记录并提交**：验收通过后，将完成内容和真实测试结果写入 `CHANGELOG.md`，将新发现且未完成的事项写入 `TODO.md`，移除其中已经完成的条目并保持清单简洁；随后只暂存本阶段文件，创建含义明确的 Git 提交，再开始下一阶段。
6. **保护共享工作区**：默认工作区可能包含用户或其他协作者的未提交修改。提交前检查 `git status` 和暂存区，只提交本阶段确认过的文件；不得覆盖、清理、回退或顺带提交无关改动。
7. **交付信息必须可核验**：阶段结束时报告完成结果、测试结论、提交哈希和仍保留的未提交文件；不能把未运行的测试、未观察的界面或推测性结果描述为已验证。
