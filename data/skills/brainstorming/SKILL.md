# 需求探索与方案收敛

把模糊想法变成能够实施和验收的方案，同时避免用过长的设计流程阻塞明确任务。

## 使用边界

- 用户只要讨论、方案或草图时，保持只读，不修改文件。
- 用户已经明确要修复、实现或替换，且范围清楚时，不强制先写设计文档或再次征求批准。
- 只对会实质改变结果的未决事项提问；可逆的小决策直接说明假设后推进。
- 一次最多提一个关键问题，不用问卷代替设计判断。

## 工作流程

1. **了解现状**：只读检查相关代码、文档、截图和已有约定，不凭文件名猜测。
2. **确认成功标准**：说清谁在什么场景下使用、要解决什么、哪些行为不能被改变。
3. **识别关键决策**：只保留会影响架构、交互、数据或验收方式的决策。
4. **比较方案**：存在真实取舍时给出 2–3 个方案，每个方案说明收益、成本和风险；没有真实分支时不凑选项。
5. **给出推荐**：明确推荐一个方案并说明原因，不把技术选择全部丢给用户。
6. **收敛验收条件**：记录范围、主要流程、异常边界、测试方式和明确不做的内容。

## 输出格式

优先给出简短的“决策摘要”：

- 目标与成功标准
- 推荐方案及取舍
- 行为边界与不做事项
- 验证方式
- 仍需用户决定的唯一关键问题（如有）

只在用户要求或项目明确要求设计文档时才写入仓库。
