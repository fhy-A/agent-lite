# 需求探索与方案收敛

把模糊想法变成能够实施和验收的方案，同时避免用过长的设计流程阻塞明确任务。

## 硬性约束

- 方案数量最多 3 个；没有真实取舍时只给 1 个。
- 读取源码只能称为“从源码确认”，不能称为“已测量”或“已验证性能”。
- 数字只有两种可以出现：源码中已有并能定位的常量，或实际命令/测试得到的测量值。
- 没有基准测试时，不预测耗时、CPU、内存、代码行数、开发工期、特定规模下的速度或优化倍数；改用定性比较和测量方法。
- 性能问题缺少基线时，第一推荐必须是补充可观测性和基准测试，不能直接推荐缓存、索引或架构改造。
- 没有基准测试时，不对实现方案排名，不使用“更快”“数量级提升”“收益明确”等性能断言；只能给出基于未来测量结果的条件分支。
- 输出前交叉检查事实与方案，不能一处确认工具会全量遍历，另一处又把同一工具描述成 O(1)。
- 关键问题必须在方案前提出；不要在长回答末尾留下空的“需要确认”标题。

## 使用边界

- 用户只要讨论、方案或草图时，保持只读，不修改文件。
- 用户已经明确要修复、实现或替换，且范围清楚时，不强制先写设计文档或再次征求批准。
- 只对会实质改变结果的未决事项提问；可逆的小决策直接说明假设后推进。
- 一次最多提一个关键问题，不用问卷代替设计判断。

## 默认探索预算

- 普通设计讨论在首次汇总前最多进行 3 次搜索或文件枚举、4 次定向读取；相邻代码应合并为一次读取。
- 找到能够回答目标、现状和主要约束的证据后立即停止调查并输出，不为追求“完整了解项目”继续遍历源码。
- 不用命令探测环境中是否存在某个可选工具，除非该事实会直接改变本次方案；性能讨论中的命令应服务于用户要求的实际基线测量。
- 达到预算仍缺少事实时，把缺口明确列为“待验证”，提出唯一关键问题或给出测量方法，不进行无边界搜索。
- 只有用户明确要求深度审计，或已发现跨模块矛盾且必须继续核实时，才能扩大预算；扩大前先说明原因和下一项要验证的事实。

## 工作流程

1. **了解现状**：只读检查相关代码、文档、截图和已有约定，不凭文件名猜测。
   - 范围明确、少量读取即可完成时直接使用声明的只读工具，不启动子任务。
   - 优先一次搜索定位入口，再读取入口及其直接依赖；不要用多个近义查询反复搜索同一实现。
   - 不要在子任务失败或超时后无说明地重复同一轮调查；保留已有结果并缩小问题。
2. **确认成功标准**：说清谁在什么场景下使用、要解决什么、哪些行为不能被改变。
3. **识别关键决策**：只保留会影响架构、交互、数据或验收方式的决策。
4. **比较方案**：存在真实取舍时给出 2–3 个方案，每个方案说明收益、成本和风险；没有真实分支时不凑选项。
5. **给出推荐**：明确推荐一个方案并说明原因，不把技术选择全部丢给用户。
6. **收敛验收条件**：记录范围、主要流程、异常边界、测试方式和明确不做的内容。

涉及性能、成本或容量时，先区分“从源码确认的行为”“实际测量结果”和“待验证假设”。没有基线、目标规模或验收阈值时，先给出测量方法或提出唯一关键问题。

讨论保持紧凑：只读取足以确认关键事实的片段；每个方案最多写“机制、适用条件、风险”三项，不重复源码清单或延伸为实施计划。

## 输出格式

优先给出简短的“决策摘要”：

- 目标与成功标准
- 推荐方案及取舍
- 行为边界与不做事项
- 验证方式
- 仍需用户决定的唯一关键问题（如有）

只在用户要求或项目明确要求设计文档时才写入仓库。

## 最终回答检查

输出前逐项删除或改写以下内容：

- 工具结果和源码都没有提供的数字、规模阈值、耗时、资源占用、优化倍数和工期。
- 没有基准测试支撑的性能排序，以及以“更快”“成本最低”“收益最大”为理由的方案推荐。
- 未读取实现却写成已确认的行为；这类内容必须移到“待验证”。
- 与工具结果矛盾的断言。若证据不足，保留条件式方案和测量方法即可，不要用常识补齐数据。
