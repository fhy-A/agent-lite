# 代码审查

默认只读。用户只要审查或报告时，不修改文件、不提交、不对外发布结果。

## 审查流程

1. **确定范围**：确认审查的 diff、提交、文件或模块，并读项目约定。用户指定范围时不扩展成全仓库扫描。
2. **理解意图**：读改动上下文、调用方和相关测试，区分新问题与已有行为。
3. **跟踪行为**：检查正常路径、错误路径、状态迁移、恢复/重试和跨模块接口。
4. **查找高影响风险**：优先安全与数据损坏，其次是功能错误、并发/性能、平台兼容和缺失测试。
5. **验证怀疑**：可用现有测试、语法检查或低风险诊断命令验证问题。不为了“有发现”而将 `TODO`、命名偏好或理论性风险当成缺陷。
6. **只报可操作问题**：每条发现都必须能说明触发条件、实际影响和修复方向。

涉及语言、标准库、框架或操作系统的边界语义时，先用当前版本文档或最小复现实验确认；不要仅凭方法名或记忆推断符号链接、glob、编码、并发等行为。

## 检查维度

- **正确性**：边界值、空值、时序、错误处理、资源释放、幂等性。
- **安全/数据**：输入验证、权限、注入、路径越界、敏感信息、破坏性操作。
- **并发/持久化**：重复请求、竞态、中断恢复、旧数据迁移。
- **性能**：只报告在实际输入规模下有明确成本的问题。
- **测试**：关键行为、错误路径和新增回归是否有有效保护。
- **可维护性**：只在复杂度会导致实际错误或阻碍当前改动时报告。

## 输出

先列发现，按 `P0` 数据/安全事故、`P1` 主要功能错误、`P2` 特定场景缺陷、`P3` 低风险改进排序。每条包含：

- 简短标题
- 精确文件与行号
- 触发场景和影响
- 建议的最小修复方向

最后给出简短范围总结和未验证风险。没有发现时明确说明未发现可操作问题，不用建议或客套填充。
