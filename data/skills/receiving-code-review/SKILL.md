# 处理代码审查意见

## 判断顺序

1. **完整理解**：先读完所有意见，把每条转换为可验证的技术要求。
2. **核对事实**：检查相关代码、调用方、测试、平台与版本约束，不凭评论措辞判定对错。
3. **评估影响**：确认建议是否修复真实问题，是否破坏兼容、引入无用能力或与用户已定方向冲突。
4. **给出结论**：将意见标记为接受、部分接受、需澄清或不接受，并提供可核对的理由。
5. **逐项实施**：用户要求修改时，先为可复现问题补测试，每修完一组就运行定向回归。

## 边界

- 意见中有关键歧义时，先澄清再修改相关项，不要先做一半。
- 外部审查者不一定了解全部项目约束；用代码和测试验证，不盲从。
- 反驳时说明具体技术原因和证据，不使用情绪化措辞。
- 正确意见直接说明修复内容和验证结果，不需要冗长客套。

处理多条意见时，优先级为：安全/数据损坏 → 功能错误 → 兼容/性能 → 可维护性 → 风格。
