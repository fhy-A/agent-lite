# 子 Agent 驱动开发

## 前置条件

- 已有明确任务列表、依赖关系和验收方式。
- 每个任务都能在一个有限文件范围内完成。
- 子 Agent 与主 Agent 共享同一项目文件，因此写入所有权必须明确。

如果任务强耦合、会频繁修改同一文件，直接由主 Agent 串行执行更安全。

## 每个任务的闭环

1. **写任务简报**：提供背景、目标、范围、现有接口、不改变项、预期测试和输出格式。
2. **委派聚焦子任务**：一个 `task` 只包含一个可验收目标。不要只发“实现计划第 3 步”这种缺少上下文的指令。
3. **先解答疑问**：子 Agent 发现关键歧义时，主 Agent 先确定边界，不让其猜测。
4. **完成实现与定向测试**：子 Agent 返回改动文件、关键决策、测试命令/结果和剩余风险。
5. **主 Agent 复核**：检查实际 diff，验证需求覆盖、项目约定、与其他任务的接口和测试有效性。
6. **处理问题**：把经过验证的问题一次性发回原子 Agent 或由主 Agent 修正，修正后重跑定向测试。
7. **阶段集成**：所有任务完成后由主 Agent 运行跨模块和全量回归，更新日志并提交。

## 并行原则

- 只读调查、无共享状态的测试分析可以并行。
- 修改同一文件、依赖未定稳接口或涉及共享迁移的任务必须串行。
- 并行任务必须显式划定可写文件；没有划定时默认只读。

## 失败处理

- 子 Agent 无法完成时，保留其已验证发现，重新缩小任务而不是重复同一模糊提示词。
- 多个子 Agent 结论冲突时，以代码、可复现测试和用户明确约束为准。
- 任何子 Agent 都不得代替用户做新的高影响决策或扩大任务范围。

不要仅根据子 Agent 的文字报告判定完成；主 Agent 必须查看实际文件并运行相关测试。
