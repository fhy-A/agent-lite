# 测试驱动开发

核心闭环是：用一个会因缺少目标行为而失败的测试定义需求，用最小实现让它通过，再在保持测试通过的前提下整理代码。

## 1. Red：写出能证明行为的失败测试

- 先读现有测试和项目命名约定。
- 一个测试只表达一个可观察行为，名称说清场景和预期结果。
- 优先通过公开接口测试真实行为，只在外部依赖无法稳定控制时使用 mock。
- 运行该测试并确认它因目标行为缺失而失败，不是因语法、路径或环境错误。
- 如果测试直接通过，先判断行为是否已存在或断言是否无效。

## 2. Green：写最小实现

- 只实现当前测试所要求的行为，不预先加选项、扩展点或无关重构。
- 再次运行定向测试，然后运行相关模块回归。
- 定向测试未通过时修正实现，不为了配合当前代码而放宽正确断言。

## 3. Refactor：在绿色状态下整理

- 消除重复，改善命名和责任边界。
- 不在重构阶段引入新行为；新行为应进入下一个 Red 循环。
- 重构后重跑定向和相关回归。

## 既有代码与缺陷

- 修复缺陷时，先写最小复现测试，确认修复前失败、修复后通过。
- 重构未有测试的旧代码时，先用特征测试锁定当前必须保留的行为。
- 如果代码难以测试，先检查接口是否过度耦合，不要用大量 mock 掩盖设计问题。

## 例外

纯文档、一次性探索原型、生成物或只有排版变化时，不强行制造低价值单元测试；改用构建、语法检查、快照或人工视觉验收，并说明验证方式。

交付时说明哪个测试曾经失败、修复了什么行为，以及最终回归结果。
