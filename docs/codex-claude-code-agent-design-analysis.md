# Codex / Claude Code Agent 设计分析报告

> 版本：v2.0 · 2026-07-22
>
> Code 实现基线：v0.5.6
>
> 用途：为后续 Agent 核心能力开发提供架构判断，不作为外部产品的逐项复刻清单。

---

## 一、结论摘要

Code v0.5.6 已经完成了最关键的架构迁移：模型调用、工具循环、授权、恢复和用量统计均由服务端持久化 `AgentRun` 承担，浏览器不再负责维持 Agent 执行生命期。因此，下一阶段不应重做运行时底座，而应集中解决下面三个缺口：

1. **运行中追加消息的语义不明确**：主任务执行时，普通新消息会被自动转换为独立后台任务，补充说明可能在主任务完成前被错误执行。
2. **委托上下文过薄**：模型 `task` 子任务和用户并行任务只得到精简提示词，缺少统一、受限、可审计的上下文契约。
3. **复杂任务缺少结构化计划**：已有并行 Child AgentRun，但缺少依赖、证据、预算和恢复语义明确的计划模型。

推荐顺序是：先实现“默认排队、显式并行”，再实现安全的 `Context Envelope`，最后增加结构化计划。Cron、Monitor、Workflow 和 worktree 隔离属于更后的自动化层。

## 二、可借鉴的外部设计

### 2.1 Codex 类设计

值得吸收的不是某个具体接口名称，而是以下约束：

- 子任务应由明确的委托动作产生，而不是通过关键词猜测用户意图。
- 子任务有独立状态、取消和资源槽位，父任务可以等待、汇总或继续推进。
- 上下文继承应是显式策略，并设置体积、安全和生命周期边界。
- 自动化任务应由持久化运行时托管，不能依赖某个浏览器标签页保持打开。

### 2.2 Claude Code 类设计

可参考的重点包括：

- 复杂任务先形成可检查的计划，再按依赖关系执行。
- 并行度、轮次、时间和成本均应有预算，而不是无限递归委托。
- 写入冲突需要显式治理；必要时才升级为工作树或工作区隔离。
- 用户始终能够看见当前阶段、阻塞原因和验证证据。

### 2.3 不应直接照搬的部分

- 不为本地 Web 产品引入 Electron 作为前置条件。
- 不重复实现一套与 `read / plan / accept / bypass` 冲突的协作模式。
- 不默认复制完整会话，也不向子任务泄露 API Key、授权 Token 或隐藏推理。
- 不用正则或关键词分类器决定消息是补充说明还是并行任务。
- 不先实现允许任意脚本执行的 Workflow DSL。

## 三、Code v0.5.6 当前架构

### 3.1 主任务 AgentRun

- 四种权限模式均由服务端 AgentRun 持续完成模型调用与工具循环。
- 浏览器负责创建任务、订阅事件、展示问卷和提交授权决定。
- AgentRun 持久化消息、事件游标、轮次、工具执行、授权、用量和终态。
- `clientRequestId` 提供创建幂等；刷新或网络重连不会重复请求上游模型。
- 服务重启后可从持久化数据恢复运行态凭据和事件订阅。

这一层已经是后续所有并行、规划和自动化能力的正确底座。

### 3.2 模型显式委托的 Child AgentRun

- 只有模型调用 `task` 工具时才创建子任务。
- 子任务继承父任务的模型、权限和工具策略，但移除 `task` 与 `request_user_input`，因此目前只支持一层委托。
- 单轮最多并发 3 个 Child AgentRun；工具结果仍按原调用顺序回填模型上下文。
- 子任务授权由父任务代理，父任务取消会向下取消；用量只向父任务合并一次。
- 当前上下文主要由专用 system prompt 与 `task.prompt` 组成，没有自动继承父会话的重要事实。

### 3.3 用户在运行期追加的消息

- 主任务仍在执行时，前端会把用户的新消息自动转成独立后台 AgentRun。
- 前端调度上限为全局 3、单会话 2，并有超时和恢复处理。
- 后台提示词只带父任务的短摘要和新请求，无法可靠区分“补充主任务”和“真正并行任务”。
- 这一通道不是模型 `task` 子任务，不能与 Child AgentRun 混为一谈。

### 3.4 已经具备的能力

| 能力 | 当前状态 |
|---|---|
| 服务端 Agent 循环 | 已完成 |
| 持久化、刷新恢复、服务重启恢复 | 已完成 |
| 授权代理与危险工具治理 | 已完成 |
| 模型显式子任务 | 已完成，一层、单轮最多 3 个 |
| 用户后台任务并发 | 已完成，全局 3 / 单会话 2 |
| 子任务独立计时和用量合并 | 已完成 |
| 运行中消息语义路由 | 待改造 |
| 统一上下文契约 | 待实现 |
| 结构化计划与预算 | 待实现 |

## 四、差距与设计决定

### 4.1 运行中消息：默认排队，显式并行

普通发送应被视为主任务的后续输入，进入当前会话的 FIFO 队列；只有用户明确选择并行入口或使用 `/parallel` 时，才创建独立后台 AgentRun。

这样可以同时满足：

- 补充说明一定能看到主任务最终结果。
- 真正独立的工作仍可并行执行。
- 不依赖不透明的意图分类器。
- 刷新后可以恢复队列并保证不会重复启动。

发送时应冻结模型、权限、思考强度和工具预设，避免真正开始执行时被后来修改的界面状态污染。

### 4.2 委托上下文：受限的 Context Envelope

模型子任务和用户显式并行任务应共享一套服务端生成的上下文信封，至少包含：

- 当前目标、父目标和执行阶段。
- 项目根目录、相关文件和已确认事实。
- 受限的会话摘要与最近可见轮次。
- 权限、可用工具、约束和期望输出。
- 独立的 Token 预算、裁剪记录和版本号。

信封必须采用字段白名单，不包含密钥、授权信息、请求载荷、隐藏推理或无关完整历史。模型提供的 `contextHint` 只能作为相关性建议，最终字段由服务端决定。

### 4.3 结构化计划：建立在现有并发底座上

计划步骤应具有稳定 ID、依赖、状态、执行者、验证证据和尝试次数；计划整体应具有最大轮次、最大子任务数和截止时间等预算。执行器复用当前 Child AgentRun 的 3 个槽位，而不是另建一套浏览器并发系统。

预算耗尽、验证失败或依赖阻塞时应暂停并向用户说明，不应无限重试。

### 4.4 自动化：后置而非前置

Cron、Monitor 与 Workflow 可以复用 AgentRun，但需要先拥有稳定的上下文、预算、取消、恢复和验证语义。初期只支持声明式动作，禁止任意 `eval`；涉及并行写入时再评估 worktree 隔离。

## 五、建议路线

| 优先级 | 阶段 | 交付结果 |
|---|---|---|
| P0 | 运行中追加消息路由 | 普通消息排队；显式入口创建并行任务；刷新可恢复 |
| P0 | Context Envelope | 模型子任务与并行任务获得安全、受限、可审计上下文 |
| P1 | 结构化规划 | 依赖、并发、预算、证据和恢复语义落到持久化计划模型 |
| P2 | 自动化能力 | Cron、Monitor、Workflow；按需加入 worktree 隔离 |

每个阶段都应独立测试、独立验收、独立提交。涉及消息时序、焦点、视觉或刷新恢复的阶段，需要前端人工验证；纯服务端结构和安全规则应由自动化测试覆盖。

## 六、关键风险

| 风险 | 处理原则 |
|---|---|
| 队列在刷新后重复启动 | 稳定 `clientRequestId`、原子状态迁移、服务端幂等创建 |
| 执行时配置与发送时不一致 | 入队时保存完整配置快照 |
| 上下文遗漏关键决定 | 结构化字段、相关轮次证据、明确降级记录 |
| 上下文泄露敏感信息 | 服务端白名单序列化与凭据拒绝测试 |
| 并行任务写同一文件 | 内容哈希/mtime 校验，冲突时暂停或串行化 |
| 计划无限循环或成本失控 | 轮次、子任务、时间和成本预算 |
