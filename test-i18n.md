
# 国际化（i18n）最佳实践摘要

## 核心原则

1. **硬编码文本与代码分离**：所有面向用户的字符串都应提取到独立的资源文件中，不应直接写在代码逻辑里。

2. **使用成熟的 i18n 库**：根据技术栈选择合适的国际化库（如 `react-intl`、`vue-i18n`、`i18next`、`gettext`），避免自研轮子。

3. **键名采用描述性命名**：使用语义化的键名（如 `"login.button.submit"`）而非直接以默认语言文本作为键。

## 关键设计

- **复数处理**：不同语言的复数规则差异巨大（阿拉伯语有 6 种复数形式），必须依赖库的复数支持而非手写 if-else。
- **日期/数字/货币格式化**：使用 `Intl` API 或库提供的格式化方法，尊重各地区的显示习惯。
- **占位符与插值**：用命名占位符而非位置占位符（`"Hello, {name}"` 优于 `"Hello, %s"`），提高可读性和翻译容错。

## 工程实践

| 实践 | 说明 |
|------|------|
| 伪本地化测试 | 用模拟翻译（加长字符、特殊符号）提前暴露布局问题 |
| CI 检查 | 在 CI 中校验所有 locale 的键是否齐全、有无遗漏 |
| 上下文注释 | 为翻译人员提供每条文本的上下文和使用场景说明 |
| 避免文本拼接 | `"Page " + n + " of " + total` 无法正确适配不同语序 |

## 注意事项

- RTL（从右到左）语言如阿拉伯语、希伯来语需要布局镜像，不仅是文本方向翻转。
- 图标和颜色在不同文化中有不同含义，需要文化敏感度审查。
- 始终为文本预留 30%-50% 的扩展空间，英语到德语或芬兰语文本长度通常增长显著。
