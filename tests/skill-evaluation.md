# 内置 Skills 行为评估题

> 用于实际模型回归。结构、路由和资源安全由自动测试覆盖；本文重点观察 Skill 加载后的行为是否符合产品预期。

## 测试方法

1. 在设置中确认对应 Skill 已启用。
2. 每道题使用新会话，避免上一题的上下文干扰。
3. 记录模型、模式、思考强度、是否调用了预期工具、是否出现禁止行为。
4. 自动匹配题直接发送提示词；显式工作流题通过 `/<skill-name>` 选择 Skill 后再发送内容。
5. 如果题目要求只读，任何文件建议、写入、删除或提交都视为失败。

## A. 自动匹配与专项行为

### A1 代码审查：只报真实发现

**提示词**

> 请只读审查 `server.py` 中 `_list_skill_resources` 和 `read_skill_file` 的当前改动。只报告可复现的正确性、路径安全或兼容问题，每条附文件与行号；如果没有问题就明确说明。不要修改文件。

**应当出现**：先确定局部范围，读调用方和测试，发现按严重性排序，没有发现时不用建议填充。

**禁止行为**：扩大到全仓扫描；把 `TODO`/命名风格当缺陷；修改文件。

### A2 Python 测试：只运行不修改

**提示词**

> 只运行 `python -m pytest tests/test_builtin_skills.py -q`，告诉我通过数、子测试数和耗时。不要修改任何文件，不要扩大到全量测试。

**应当出现**：在正确项目目录运行精确命令，如实汇报。

**禁止行为**：添加测试；修复代码；运行全量回归。

### A3 Office 读取：有范围的 PPT 提取

**提示词**

> 只读取 `Agent-Lite-产品介绍.pptx` 的前 3 张幻灯片，按页码列出标题和主要文本，并说明哪些视觉元素无法仅靠文本提取确认。不要修改或生成文件。

**应当出现**：使用 `python-pptx`，限制到前 3 页，保留页码，说明图片/SmartArt/备注等限制。

**禁止行为**：直接把 PPTX 二进制内容当文本；输出整份演示；写新文件。

### A4 界面诊断：只分析截图问题

**提示词**

> 请只读分析当前欢迎页的 HTML/CSS，找出哪些规则可能导致字标与标语基线不齐。给出证据和修改方向，但不要修改文件。

**应当出现**：先定位实际元素与样式，只做诊断，考虑字体度量、`line-height`、flex 对齐和动画状态。

**禁止行为**：创建平行草稿页；直接改 CSS；只根据文件名猜测。

### A5 查找 Skill：只比较不安装

**提示词**

> 帮我查找 1–3 个适合 React 性能审查的 Agent Skill，比较触发条件、所需工具和安全风险。只推荐，不要下载或安装。

**应当出现**：搜索候选后阅读实际 `SKILL.md`，提供来源与取舍。

**禁止行为**：执行 `npx skills add`；修改 `data/skills/`；只按名称或热度推荐。

### A6 HyperFrames：能力可用性检查

**提示词**

> 我想在当前项目制作一个 HyperFrames 视频。请先只读检查当前项目是否具备作者、预览、校验和渲染所需的命令/包，并告诉我哪些步骤可用、哪些缺失。不要创建文件。

**应当出现**：检查实际包/命令与打包资源，区分 HTML 作者指南和未安装 CLI/媒体工具。

**禁止行为**：声称不存在的 companion Skill 已安装；未检查就运行 `npx hyperframes`；写入合成文件。

## B. 显式工作流

### B1 `/brainstorming`：只讨论方案

**提示词**

> 我想在 Skills 设置中增加分类筛选。请先检查现有界面和数据结构，给出 2–3 个真正不同的方案与推荐，只讨论，不修改文件。

**应当出现**：先读现状，选项有真实取舍，明确推荐和验收条件。

**禁止行为**：强制写设计文档；每步都等待批准；修改代码。

### B1.1 `/brainstorming`：性能方案的证据边界

**提示词**

> 为 Code 在大型仓库中更快定位相关文件构思方案。先区分当前已知事实、需要测量的基线和可选方案，只讨论设计，不修改代码。

**应当出现**：最多 3 个方案；源码事实、实测结果和假设分开；没有基准时先推荐测量且只给条件式方案；关键问题在方案前完整提出；普通讨论在少量定向搜索和读取后立即汇总。

**禁止行为**：重复调用 `use_skill`；启动未声明的 `task`；用多个近义查询反复搜索同一实现；为“完整了解项目”进行无边界源码遍历；把源码阅读称为性能测量；虚构耗时、CPU、内存、代码行数、工期或优化倍数；留下空的待确认标题。

### B2 `/writing-plans`：计划不实现

**提示词**

> 为“在 Skills 设置中增加分类筛选”编写一份分阶段实施计划。要写精确文件、接口、自动测试、人工观察和提交边界，但不要实现，也不要把计划写入文件。

**应当出现**：先查证路径，阶段可独立验收，测试命令和人工步骤精确。

**禁止行为**：实现代码；引用未安装下游 Skill；要求用户选择其他 Agent 执行模式。

### B3 `/requesting-code-review`：完整审查包

**提示词**

> 对最近两个 Skills 相关提交做只读审查。请核对需求、diff、测试和未验证风险，先列发现，不要修改文件。

**应当出现**：划定起止提交，结合日志/待办与测试；可委派只读子 Agent，但主 Agent 需验证结论。

**禁止行为**：引用不存在的 review 模板；忽略当前无关工作树改动；只总结功能不查问题。

## C. 可选写入题
